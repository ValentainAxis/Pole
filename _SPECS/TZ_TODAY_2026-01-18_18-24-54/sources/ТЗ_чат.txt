ТЕХНИЧЕСКОЕ ЗАДАНИЕ: GENESIS — Студия наблюдения и исполнения
Сводный документ (integrated) v2.0-rc1
Статус: Модули I–VI согласованы; Resonance Protocol v1.1 и Observation Shell v1.1 финализированы; MVP Observation Shell v0.1-r1 утверждён как «датчик правды». Реализация начинается после полевого аудита среды и фиксации модуля развёртывания/эксплуатации (Deployment & Field Ops).

МОДУЛЬ I — ВВЕДЕНИЕ (v1.1)


Назначение системы
Система предназначена для преобразования задач пользователя в последовательность исполняемых действий в вычислительной среде GENESIS с полной наблюдаемостью процесса и проверяемыми следами выполнения.


Система обеспечивает:
— интерпретацию намерений пользователя Ассистентом;
— формирование плана (последовательность Steps);
— передачу шагов исполнителю Agent Runa через единый протокол (Resonance Protocol);
— фиксацию следов выполнения в GENESIS (TRACES);
— визуальное наблюдение состояния выполнения через Observation Shell (read-only).


Границы системы


2.1. Система включает
— диалоговый контур Пользователь ↔ Ассистент (формирование intent, планирование, контрольные точки);
— управляющий контур Ассистент ↔ Agent Runa через Resonance Protocol;
— протокол взаимодействия Resonance Protocol (единственный канал связи Ассистент ↔ Agent Runa);
— интерфейс наблюдения Observation Shell (орган наблюдаемости, read-only);
— вычисляемую фазу выполнения (computed_phase), определяемую Observation Shell по потоку Status/Result.
2.2. Система не включает
— дополнительных каналов связи между Ассистентом и Agent Runa (логи, стримы, «быстрые» туннели, сигнальные потоки);
— принятия стратегических решений исполнителем (Agent Runa);
— скрытых действий, не отображаемых наблюдаемым контуром (Status/Result + TRACES);
— управления выполнением через Observation Shell (Shell не управляет, не подтверждает, не отменяет).
Примечание об эксплуатации: система не требует постоянной ручной работы оператора в терминале для наблюдения и понимания состояния выполнения; эксплуатационные действия среды (запуск/останов сервисов, аудит) относятся к уровню GENESIS и не являются каналом управления шагами.
2.3. Внешние зависимости
Система опирается на GENESIS — вычислительную и файловую инфраструктуру, включающую:
— VAULT как хранилище следов;
— _TRACES как каталог фиксации фактов выполнения;
— bin как каталог утверждённых исполняемых утилит;
— системные инструменты Linux в пределах политик.
GENESIS является средой выполнения и фиксации фактов и рассматривается как внешний фундаментальный слой.


Целевые роли


3.1. Пользователь
— формулирует задачу в диалоге;
— принимает решения в контрольных точках (confirmation points) на уровне Ассистента;
— получает итог и наблюдает процесс через Shell.
Пользователь не является исполнителем шагов и не взаимодействует с протоколом.
3.2. Ассистент
— извлекает intent из диалога;
— строит plan (последовательность Steps);
— определяет контрольные точки и получает решения пользователя;
— передаёт шаги в Agent Runa по Resonance Protocol;
— агрегирует итог и формирует Result.
3.3. Agent Runa (Помощник-исполнитель)
— выполняет шаги плана детерминированно в рамках Workspace;
— применяет политики безопасности исполнения (allowlist, FS, network);
— фиксирует следы (TRACES);
— возвращает статусы исполнения (Status) и ошибки через протокол.
Agent Runa не принимает стратегических решений и не меняет план.


Ключевые принципы


4.1. Разделение уровней
— намерение: Пользователь;
— управление: Ассистент;
— исполнение: Agent Runa.
Смешение уровней запрещено.
4.2. Наблюдаемость
Каждый значимый этап исполнения представлен наблюдаемым контуром:
— Status/Result в протоколе;
— TRACES в VAULT.
Скрытые операции запрещены как модель поведения системы.
4.3. Управляемая детализация
Технические детали по умолчанию скрыты; детализация доступна в read-only режиме через Observation Shell и ограничена структурированными полями (без неконтролируемого “сыра” stdout/stderr).
4.4. Контрольные точки подтверждения
Контрольные точки (confirmation points) реализуются на уровне Ассистента и Пользователя. Они не являются состояниями исполнения Agent Runa и не отражаются в Step/Status как логика ожидания подтверждения.
4.5. Проверяемость результата
Факт выполнения определяется только наблюдаемыми артефактами:
— Status(success|error|aborted) в протоколе;
— валидный Trace в _TRACES (если шаг завершён и должен оставлять след по политике).

МОДУЛЬ II — ТЕРМИНЫ И РОЛИ (v1.1)
2.1. Основные понятия
Система: комплекс взаимодействующих компонентов (Ассистент, Agent Runa, Protocol, Shell), реализующий разделение уровней и наблюдаемость.
GENESIS: внешняя инфраструктура выполнения и фиксации фактов (Workspace, VAULT, _TRACES, bin, конфиги).
Пользователь: источник задач и критериев приемки; принимает решения в контрольных точках.
Ассистент: управляющий компонент (intent → plan → контроль → Result).
Agent Runa: исполнительный компонент, выполняющий Steps, применяющий политики безопасности исполнения, фиксирующий TRACES.
Resonance Protocol: единственный канал связи Ассистент ↔ Agent Runa; определяет формат и правила обмена сообщениями.
Observation Shell: орган наблюдаемости (read-only), отображающий реальность выполнения на основе Protocol + TRACES.
computed_phase: вычисляемое состояние сессии в Shell (planned|executing|finalizing|done), определяемое детерминированно по Status/Result.
Intent: формализованное представление задачи, извлечённое Ассистентом.
Plan: последовательность Steps, исполняемая Agent Runa.
Step: минимальная единица исполнения плана.
Status: структурированное сообщение о состоянии исполнения шага.
Result: итог сессии исполнения, сформированный Ассистентом.
Trace: структурированный след выполнения, записанный в VAULT/_TRACES.
2.2. Контуры
Диалоговый контур: Пользователь ↔ Ассистент.
Исполнительный контур: Ассистент ↔ Protocol ↔ Agent Runa.
Наблюдательный контур: Observation Shell читает Protocol (read-only) + TRACES (read-only).
2.3. Идентификаторы
session_id: идентификатор сессии исполнения.
workspace_id: в версии 1.1 тождествен session_id; возможность разведения зарезервирована для будущих версий; вычисление/связь при разведении будет определяться Resonance Protocol.

МОДУЛЬ III — ФУНДАМЕНТ: GENESIS (v1.2)
3.1. Назначение фундаментального слоя
GENESIS предоставляет:
— детерминированную среду выполнения для Agent Runa;
— хранилище проверяемых следов выполнения (VAULT/_TRACES);
— набор утверждённых исполняемых процедур (GENESIS/bin);
— базовые политики безопасности на уровне пользовательской среды.
3.2. Workspace
3.2.1. Определение
Workspace — изолированная файловая область в пределах GENESIS, используемая для выполнения одной сессии plan.
3.2.2. Путь
~/GENESIS/workspaces/<session_id>/
3.2.3. Жизненный цикл
Создание: выполняется Agent Runa по команде Ассистента перед первым шагом.
Использование: все действия по шагам выполняются внутри Workspace.
Очистка: отложенно по политике хранения или по служебной команде.
3.2.4. Инварианты
— замкнутость: отсутствие доступа вне Workspace (кроме read-only конфигов и целевых путей, разрешённых политикой);
— воспроизводимость: ключевые факты фиксируются;
— наблюдаемость: результаты доступны через TRACES.
3.3. Execution Allowlist
3.3.1. Общая модель
Agent Runa исполняет только утверждённые исполняемые файлы.
3.3.2. Хранение
~/GENESIS/config/allowlist.yml (read-only для Agent Runa).
3.3.3. Scope
Allowlist — список разрешённых исполняемых файлов (включая обёртки в ~/GENESIS/bin). Валидация параметров делегируется обёрткам в bin/ и/или политике безопасности, но не является обязанностью allowlist.
3.4. Конфигурация политик безопасности
~/GENESIS/config/security_policies.yml (read-only для Agent Runa).
Формат определяется модулем «Конфигурация и развёртывание» (вне текущего набора I–VI).
3.5. TRACES
3.5.1. Путь
~/GENESIS/VAULT/_TRACES/
3.5.2. Формат Trace (минимальный, допускает расширения)
{
"id": "<trace_id>",
"timestamp": "<iso8601>",
"context": {
"session_id": "<session_id>",
"step_id": "<step_id>"
},
"action": "<description>",
"status": "success" | "error" | "aborted",
"details": { ... },
"output": { ... },
"links": {
"previous": "<trace_id|null>"
}
}
3.5.3. Связность
Для линейных планов порядок восстанавливается детерминированно по links.previous и timestamp.
3.6. Ограничения
— выполнение зависит от состояния ОС и ресурсов;
— изоляция реализуется на уровне пользовательской среды/политик;
— Agent Runa не «лечит» среду, а фиксирует ошибки как environment_error.
3.7. Безопасность (базовые правила)
Разрешено: работа в Workspace; запуск allowlist-команд; чтение конфигов.
Запрещено по умолчанию: произвольные бинарники; выход за пределы разрешённых путей; сетевые операции (если не разрешены политикой).
3.7.3. Атомарность и компенсация
Каждый Step атомарен. Автоматический откат не выполняется. Восстановление после ошибок реализуется через отдельные компенсационные шаги (type="compensation"), планируемые Ассистентом и исполняемые Agent Runa как обычные Step.

МОДУЛЬ IV — ИСПОЛНИТЕЛЬНЫЙ КОМПОНЕНТ: AGENT RUNA (v1.1)
4.1. Назначение
Agent Runa — исполнительный механизм, обеспечивающий строгое и предсказуемое выполнение шагов плана в рамках Workspace и политик GENESIS.
4.2. Контракт исполнения
4.2.1. Вход: Step object (в протоколе)
{
"id": "<uuid>",
"type": "action" | "compensation",
"command": "<string>",
"metadata": { ... },
"session_id": "<string>",
"timestamp_issued_ms": <unix_ms>
}
4.2.2. Выход: Status (в протоколе)
{
"step_id": "<uuid>",
"session_id": "<string>",
"state": "pending" | "running" | "success" | "error" | "aborted",
"error_type": null
| "execution_error"
| "security_violation"
| "environment_error"
| "protocol_error"
| "user_abort",
"diagnostic": { ... },
"timestamp_ms": <unix_ms>
}
4.3. Модель исполнения
— валидация команды по allowlist;
— проверка политик безопасности (FS, network, пр.);
— подготовка/проверка Workspace;
— исполнение;
— сбор результата;
— запись Trace;
— возврат Status.
4.3.2. Компенсационный шаг
Compensation — отдельный Step (type="compensation"), планируемый Ассистентом. Agent Runa исполняет его как обычный шаг.
4.4. Workspace
Создание Workspace инициируется Ассистентом и выполняется Agent Runa по служебной команде/старту сессии перед первым шагом.
4.5. Allowlist
Allowlist ограничивается списком исполняемых файлов. Проверка параметров — зона ответственности обёрток в ~/GENESIS/bin и/или политик безопасности.
4.6. TRACES
Каждый шаг фиксируется как Trace по политике фиксации. Связность линейного плана восстанавливается детерминированно.
4.7. Ошибки
Ошибки не запускают автоматический откат. Решение о компенсации принимает Ассистент.
4.8. Недопустимые действия
— выход за пределы разрешённой области;
— исполнение вне allowlist;
— самовольный сетевой доступ;
— изменение плана;
— любые скрытые действия, минующие наблюдаемый контур.
4.9. НФТ
— детерминированность исполнения (одинаковые входы → одинаковые наблюдаемые эффекты в пределах среды);
— устойчивость под ограниченными ресурсами;
— низкая латентность Step → Status (метрики задаются отдельно).
4.10. Жизненный цикл
Agent Runa — долгоживущий сервис. Единственный канал взаимодействия с Ассистентом: Resonance Protocol. Конфиги allowlist.yml и security_policies.yml загружаются при инициализации.

МОДУЛЬ V — RESONANCE PROTOCOL (v1.1)
5.1. Назначение
Resonance Protocol является единственным каналом связи между Ассистентом и Agent Runa (см. Модуль IV, 4.10). Протокол определяет формат, последовательность и правила обмена сообщениями и обеспечивает целостность обмена.
5.2. Принципы
— транспортная нейтральность: протокол определяет модель сообщений, не привязываясь к транспорту;
— логическая синхронность: request → ack → поток status/result (транспорт может быть асинхронным);
— контрактная неизменность: Step/Status соответствуют Модулю IV; изменения только через версионирование;
— один канал: любые дополнительные каналы связи Ассистент ↔ Agent Runa запрещены. Расширения допускаются только через новую явно версионированную спецификацию протокола.
5.3. Модель сессии
session_id — глобальный идентификатор сессии.
workspace_id — в v1.1 строго равен session_id; возможность разведения зарезервирована.
5.3.2. session_init (служебное сообщение)
{
"type": "service",
"service": "session_init",
"session_id": "<string>",
"timestamp_ms": <unix_ms>,
"protocol_version": "1.1",
"plan_name": "<optional string>",
"step_count_total": <optional int>
}
5.4. Входящие сообщения
5.4.1. Envelope
{
"type": "step" | "service",
"session_id": "<string>",
"payload": { ... }
}
5.4.2. Step payload (из Модуля IV)
{
"id": "<string>",
"type": "action" | "compensation",
"command": "<string>",
"metadata": { ... },
"session_id": "<string>",
"timestamp_issued_ms": <unix_ms>
}
5.4.3. Service messages (фиксированные структуры)
ping:     { "type":"service", "service":"ping", "session_id":"<string>", "timestamp_ms":<unix_ms> }
cancel:   { "type":"service", "service":"cancel", "session_id":"<string>", "reason":"<string>", "timestamp_ms":<unix_ms> }
shutdown: { "type":"service", "service":"shutdown", "timestamp_ms":<unix_ms> }
5.5. Исходящие сообщения
5.5.1. Ack
{
"type": "ack",
"step_id": "<string>",
"session_id": "<string>",
"timestamp_received_ms": <unix_ms>
}
5.5.2. Status (из Модуля IV)
{
"type": "status",
"session_id": "<string>",
"step_id": "<string>",
"state": "pending" | "running" | "success" | "error" | "aborted",
"error_type": null | "execution_error" | "security_violation" | "environment_error" | "protocol_error" | "user_abort",
"timestamp_ms": <unix_ms>,
"diagnostic": { ... }
}
5.5.3. Result
{
"type": "result",
"session_id": "<string>",
"summary": "<string>",
"timestamp_ms": <unix_ms>
}
5.5.4. Protocol Error
{
"type": "protocol_error",
"session_id": "<string|null>",
"received_type": "<string|null>",
"received_session_id": "<string|null>",
"error_code": "invalid_format" | "unknown_type" | "version_mismatch" | "session_not_found",
"timestamp_ms": <unix_ms>
}
5.6. Модель переходов (для шага)
pending → running → success
pending → running → error
pending → running → aborted
compensation — это отдельный Step, а не состояние.
5.7. Правила обработки
— любой Step должен вызвать Ack;
— каждый Status должен соответствовать ранее принятому Step;
— Result в сессии ровно один.
5.8. Безопасность (граница ответственности)
Протокол обеспечивает только: целостность, валидность формата, соответствие session_id. Политики безопасности исполнения (allowlist, FS, network) применяет Agent Runa.
5.9. Ограничения
— нет прямой передачи stdout/stderr;
— нет скрытых каналов;
— планы линейные в v1.1.
5.10. НФТ
Требования к латентности относятся к выбранному транспортному уровню; количественные цели задаются в отдельном модуле NFR/приёмки.

МОДУЛЬ VI — OBSERVATION SHELL (v1.1)
6.1. Назначение
Observation Shell — орган визуальной наблюдаемости GENESIS. Он не управляет, не вмешивается, не инициирует действий. Он отображает реальность выполнения на основе Protocol + TRACES.
6.2. Принципы
6.2.1. Пассивность
Shell не отправляет команд и не влияет на выполнение.
6.2.2. Визуальная консистенция
Интерфейс обновляется только в ответ на изменения входных данных. Переходы/анимации допускаются только с фиксированными параметрами и без семантической нагрузки.
6.2.3. Источники истины
Shell отображает только:
— поток Protocol сообщений (session_init, ack, status, result) в режиме read-only;
— VAULT/_TRACES в режиме read-only.
6.3. Источники данных
— Protocol поток (read-only);
— TRACES (read-only), доступ по session_id;
— session_init метаданные (опциональные plan_name, step_count_total).
6.4. computed_phase: функция phase(session)
planned: session_id известен, Status ещё нет
executing: получен Status(state=running)
finalizing: все шаги в terminal state (success/error/aborted), Result ещё нет
done: получен Result
6.5. Каноническая сцена интерфейса
6.5.1. Заголовок
— session_id
— computed_phase
— plan_name (если есть, иначе “—”)
— step_count_total (если есть, иначе “—”)
— timestamp_started (из session_init.timestamp_ms)
6.5.2. Лист шагов
Для каждого шага:
— step_id
— state
— command_name (базовое имя команды без параметров)
— timestamp_last (последний timestamp_ms из Status)
— trace_available (bool)
6.5.3. Общий прогресс
Если step_count_total известно:
progress = (completed_steps / step_count_total) * 100%
Если неизвестно: progress не вычисляется (отображается “—”).
6.5.4. Управляемая детализация
По клику на шаг: Trace View (структурированные поля Trace), без доступа к неструктурированному сырому stdout/stderr.
6.5.5. Активная сессия и архив
Shell отображает одну активную сессию. Просмотр завершённых сессий реализуется как архивный режим (read-only), не нарушающий принцип одного центра тяжести для активной задачи.
6.6. Запреты
— Shell не отправляет сообщений в Protocol;
— Shell не запрашивает данные у Ассистента как у сервиса;
— Shell не открывает прямых каналов к Agent Runa;
— Shell не создаёт дополнительных каналов обмена.
6.7. Ограничения
— stdout/stderr не отображаются;
— нелинейные графы выполнения зарезервированы для будущих версий;
— управление/отмена/подтверждения отсутствуют.
6.8. Fallback режим
При недоступности GUI Shell выводит структурированный текст: session_id, computed_phase, список steps{id,state,timestamp}.
6.9. НФТ
— обновление интерфейса не чаще одного раза в 100 мс;
— устойчивость при высокой частоте Status;
— прозрачная деградация до fallback режима.

ПРИЛОЖЕНИЕ A — GENESIS EXECUTION LOOP (v0.1)
Контур выполнения
[User Intent] → [Assistant] → [Resonance Protocol] → [Agent Runa] → [TRACES] → [Observation Shell]
Инварианты (правила поля)
— детерминированная цепочка: Step → Ack → Status(running) → Status(terminal) → Trace → Result
— любая потеря звена = наблюдаемая аномалия
— единственный канал: только Resonance Protocol между Ассистентом и Agent Runa
— Shell read-only: видит, но не влияет
— факт выполнения определяется Status + Trace (по политике фиксации)

ПРИЛОЖЕНИЕ B — MVP OBSERVATION SHELL “ДАТЧИК ПРАВДЫ” (v0.1-r1)
B.1. Назначение
MVP Shell — минимальный орган верификации реальности выполнения. Он показывает активную сессию, шаги, состояния, вычисляемую фазу и связанный Trace (по запросу). Управление отсутствует.
B.2. Источники данных
— Protocol поток (read-only): session_init, status, result (ack допускается отображать как технический сигнал, но Shell не участвует в ack)
— TRACES (read-only): VAULT/_TRACES по session_id
B.3. Поведение
— отображать одну активную сессию
— вычислять computed_phase детерминированно (см. Модуль VI)
— показывать список шагов и их terminal state
— по клику на шаг открывать структурированный Trace View (если доступен)
B.4. Намеренно отсутствует
— любые элементы управления (start/cancel/confirm)
— любые дополнительные каналы к Agent Runa
— отображение stdout/stderr
— нелинейные графы
— «умные» интерпретации результата
B.5. Критерии “датчика правды”
— Shell отображает то же состояние, что проходит по Protocol, без подмен
— Trace View показывает только структурированные поля
— отсутствие Shell не влияет на выполнение; присутствие Shell не меняет выполнение

ИТОГ
Этот интегрированный документ v2.0-rc1 фиксирует систему как архитектуру поля: единственный канал исполнения (Protocol), строгий исполнитель (Runa), наблюдаемость и проверяемость (TRACES + Shell), и минимальный MVP как “датчик правды”. Дальнейшая эволюция допустима только через версионирование модулей и сохранение инвариантов единственного канала и разделения уровней.
